| TD-023 | S6-5 | security | GitHub Dependabot flags `ecdsa==0.19.2` (transitive dep of `python-jose`, pulled in regardless of the `[cryptography]` extra) for GHSA-wj6h-64fc-37mp / CVE-2024-23342 ("Minerva timing attack on P-256", HIGH). Upstream maintainer has stated there is no planned fix — side-channel attacks are out of scope for pure-Python `ecdsa`. The app only uses HS256 (HMAC, symmetric) for JWTs (`app/core/config.py`, `deps.py`, `security.py`), so the vulnerable ECDSA signing/ECDH code paths in `ecdsa` are never exercised. Note: `pip-audit` (added in S6-5) does **not** flag this — the OSV advisory's `ranges` say all versions are affected with no `fixed` event, but its explicit `versions` enumeration stops at `0.19.1` (does not list `0.19.2`), so pip-audit's version match misses it. Real-world example of pip-audit/OSV and GitHub Dependabot/GHSA disagreeing on the same advisory. Fix candidates: (1) dismiss the Dependabot alert via GitHub UI with reason "vulnerable code is not used"; (2) longer-term, evaluate migrating `python-jose` → `PyJWT` (no `ecdsa` dependency) if JWT handling is revisited. | Medium | S6-5 |
| TD-024 | S6-8 | data-integrity | `create_transaction` (`app/services/transaction_service.py`) validates that all `entries[]` share the same `currency`, but never checks that `currency` against `Account.currency`. `calculate_balance` (`app/services/balance.py`) sums raw `Entry.amount` (minor units in `entry.currency`) without any currency filter, so posting entries whose currency differs from the account's `currency` column produces a balance that mixes minor units across currencies — numerically meaningless. Discovered during S6-8 load-test prep when considering reuse of existing EUR accounts (1100/2000) with `locustfile.py`'s fixed `currency: "USD"` entries; avoided by creating dedicated USD accounts for the load test instead. Fix candidates: (1) validate `entry.currency == account.currency` in `create_transaction` (422 on mismatch); (2) or have `calculate_balance` sum `converted_amount_usd` instead of raw `amount` to support genuinely multi-currency accounts. | Medium | S6-8 |
| TD-025 | S6-8 | api-design | `list_accounts` (`app/api/v1/routes/accounts.py`) and `list_transactions` (`app/api/v1/routes/transactions.py`) execute `select(...)` without `.order_by(...)`, so PostgreSQL does not guarantee row order. `list_accounts` has no pagination, but clients that assume a stable order (e.g. `locustfile.py`'s `on_start`, which originally took `accounts[0]`/`accounts[1]`) get non-deterministic results — discovered in S6-8 when newly-created USD accounts did not sort before pre-existing EUR accounts. `list_transactions` is paginated via `limit`/`offset`, which is the more serious case: `LIMIT`/`OFFSET` without `ORDER BY` can return duplicate or skipped rows across pages under concurrent writes — a pagination-correctness bug. `get_ledger_entries` (`app/services/ledger_service.py`) and `get_audit_logs` already use `order_by` correctly and are unaffected. Fix: add `.order_by(Account.code)` to `list_accounts` and `.order_by(Transaction.transaction_date.desc(), Transaction.id)` (or similar) to `list_transactions`. | Medium | S6-8 |
