| TD-015 | S5-8 | performance | `GET /accounts/{id}/balance` end-to-end latency on a cache hit is ~99ms (curl `time_total`), far above the "cache hit < 10ms" DONE-condition reading. Isolating the layers showed Redis itself responds in well under 1ms (`docker compose exec redis redis-cli --latency` ‚Üí `avg: 0.93`), so the ~10ms target is met *for the cache lookup itself* ‚Äî the remaining ~98ms is HTTP/auth/serialization overhead, most likely `get_current_user` (`app/core/deps.py`) re-querying the `users` table on every authenticated request to resolve the JWT subject into a `User` row. A secondary contributor is that `get_redis_client` (and `get_redis` in idempotency.py) creates a new ConnectionPool on every request via `aioredis.from_url()`, discarding pooling benefits. Fix candidates: embed role/active-status claims in JWT; lifespan-scoped Redis client (see TD-020); short-lived in-process cache for `get_current_user`. | Low | S5-8 |
| TD-016 | review | models | `Entry.amount` is mapped as `Integer` (32-bit), but all documentation and `converted_amount_usd` use `BigInteger`. 32-bit overflow at ~2,147,483,647 minor units ‚âà $21.4M (cents), which is reachable in a real payment ledger. Fix: change `Entry.amount` mapping to `BigInteger` and generate an Alembic migration. | High | review |
| TD-017 | review | idempotency | Idempotency key is `SET NX` in the dependency *before* the DB transaction commits (`get_db` commits after the route handler returns). If `create_transaction` raises 422 or the commit fails, the Redis key persists for 24h and blocks all legitimate retries. Fix: either delete the key on failure (exception-safe generator dependency) or set the key only after a confirmed commit. Relates to TD-004/TD-005. | Medium | review |
| TD-018 | review | consistency | Balance cache invalidation (`redis.delete`) in `post_transaction` runs inside the route handler, before `get_db`'s auto-commit executes. A concurrent `GET /balance` arriving between the delete and commit will re-compute against pre-commit data and re-cache a stale balance for the full TTL (60s). Fix: call `await db.commit()` explicitly in the route before issuing `redis.delete`. | Medium | review |
| TD-019 | review | architecture | Service layer (`transaction_service`, `currency_service`, `user_service`) raises `fastapi.HTTPException` directly, coupling domain logic to the HTTP transport. Services cannot be reused or unit-tested without a FastAPI context. Fix: raise domain-specific exceptions in services; map to HTTP status codes in an exception handler or at the route layer. Sub-item of TD-008. | Medium | review |
| TD-020 | review | infra/async | `core/cache.py:get_redis_client` and `dependencies/idempotency.py:get_redis` are near-duplicate functions that both call `aioredis.from_url()` on every request, creating and tearing down a `ConnectionPool` per request. Fix: create a single `aioredis.Redis` client in the `lifespan` context, store on `app.state`, and inject via a lightweight dependency. Contributes to TD-015 latency. | Medium | review |
| TD-021 | review | audit | `POST /currencies`, `POST /exchange-rates`, and `POST /users` do not call `log_action`. Admin-mutating operations are invisible to the audit trail. Fix: add `log_action` calls consistent with the pattern in `create_account` and `create_transaction`. | Medium | review |
| TD-022 | review | architecture | `accounts.py:create_account` and `audit_logs.py:get_audit_logs` embed ORM query logic and `log_action` calls directly in the route handler, bypassing the service layer that other endpoints use. The 3-layer rule (api ‚Üí services ‚Üí models) is unevenly applied. Fix: extract account creation and audit-log query into service functions. | Low | review |
