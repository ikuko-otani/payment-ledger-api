| TD-026 | S6-8 | performance | At 100 concurrent locust users (`-u 100 -r 10 -t 60s`), `POST /api/v1/auth/login` (the first DB-backed call in `on_start`) saturates the async SQLAlchemy engine's connection pool. `app/db/session.py`'s `create_async_engine(...)` does not set `pool_size`/`max_overflow`, so SQLAlchemy defaults apply (`pool_size=5`, `max_overflow=10` → 15 concurrent connections max, `pool_timeout=30s`). With 100 users ramping up in 10s, most `/auth/login` requests queue for a connection; 2/41 requests exceeded the 30s `pool_timeout` and failed with `sqlalchemy.exc.TimeoutError: QueuePool limit of size 5 overflow 10 reached` (HTTP 500). Result: p99 latency ~48s (target: <100ms), 4.88% error rate. PostgreSQL `max_connections=100` (confirmed via `SHOW max_connections`) leaves ample headroom for a larger pool. Fix: set `pool_size`/`max_overflow` explicitly in `create_async_engine(...)`. **Resolved**: `pool_size=20, max_overflow=30` applied to `app/db/session.py`. Re-running the identical 100-user test confirms the fix: error rate dropped from 4.88% to 0%. However, this exposed a deeper bottleneck — see TD-027. Baseline (before fix) and confirmation (after fix) data: `docs/loadtest/result_100users_*.csv`. | High | S6-8 |
| TD-027 | S6-8 | performance | `verify_password` (`app/core/security.py:18`, wraps `bcrypt.checkpw(...)`) is a synchronous, CPU-bound call invoked directly — without `await asyncio.to_thread(...)` or `run_in_executor` — inside `async def login` (`app/api/v1/routes/auth.py:24`). Each bcrypt check blocks FastAPI's single event loop for its full duration, so ALL concurrent requests across the whole app (not just other logins) queue behind it. Discovered immediately after the TD-026 `pool_size`/`max_overflow` fix: with the connection-pool ceiling removed, all 100 concurrent `/auth/login` requests now succeed (0% error rate, down from 4.88%), but they queue behind each other for bcrypt — observed latency: min 5.2s, median 27s, max 51.6s, aggregated p99 ≈ 52s (target: <100ms). Net effect of TD-026's fix: errors eliminated, but raw p99 latency got slightly worse (48s → 52s) because more requests now survive long enough to join the bcrypt queue instead of failing early on a pool timeout. Only 9/100 simulated users got past `on_start` to execute a `@task` within the 60s window — `post_transaction` (weight 7) recorded zero requests entirely. **Resolved**: `verify_password`/`get_password_hash` (`app/core/security.py`) now wrap the bcrypt calls in `await asyncio.to_thread(...)`; call sites in `app/api/v1/routes/auth.py`, `app/services/user_service.py`, `tests/conftest.py` updated to `await`. Re-running the 100-user test shows only marginal change (login p99 52s → 50s, total reqs 118 → 145) — see TD-028 for why the impact was small. Baseline data: `docs/loadtest/result_100users_*.csv`. | High | S6-8 |
