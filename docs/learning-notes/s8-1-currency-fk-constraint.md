# S8-1: accounts/entries.currency → currencies.code FK Constraint

**Date**: 2026-06-19
**Branch**: feature/s8-1-currency-fk-constraint (merged to main)
**Goal**: Close TD-035 — add `ForeignKey("currencies.code")` to `accounts.currency`
and `entries.currency`, sealing the referential integrity gap discovered in S7-6.

---

## Step C Walkthrough Summary

### What was implemented

**1. Alembic migration** (`20260618_1856_d117489c7a4b_add_currency_fk_to_accounts_entries.py`)

Added FK constraints from `accounts.currency` and `entries.currency` to
`currencies.code`. The migration includes a data-integrity pre-check in
`upgrade()` using `op.get_bind()` to count orphaned currency codes before
applying the constraints.

**2. ORM model updates** (`app/models/account.py`, `app/models/entry.py`)

Added `ForeignKey("currencies.code")` to the `currency` mapped column in both
models. The correct argument order is `String(3)` first, then `ForeignKey(...)` —
reversed order raises `ArgumentError`.

**3. conftest.py seed** (`tests/conftest.py`)

Added USD/EUR/JPY seed rows in `clean_db` after the pre-test TRUNCATE. This
ensures all existing tests that create accounts with these currencies continue
to work without per-test changes.

**4. Application-layer validation** (`app/services/account_service.py`,
`app/api/v1/routes/accounts.py`)

Added `currency_repo.find_by_code(payload.currency)` check before the account
INSERT, raising `ValidationError` (422) on unknown currency codes. This gives
users a clear error message instead of a 500 IntegrityError from the DB.
`CurrencyRepository` was injected into the `create_account` route via a new
`CurrencyRepoDep` annotated dependency.

**5. Test fixes** (6 test files)

Tests that created USD/EUR/JPY currencies directly conflicted with the new
seed. Four fix patterns were applied:

| Pattern | Files | Fix |
|---------|-------|-----|
| HTTP POST with non-seeded code | test_currencies, test_audit_log | Switch codes to GBP/CHF |
| ORM INSERT Currency | test_ledger, test_transactions, test_transactions_multi_currency | SELECT existing instead of INSERT |
| HTTP POST with id dependency | test_currency_conversion | `_seed_currency` GET-first, then POST if absent |
| Seeding loop now redundant | test_transactions_multi_currency | Remove seeding; rely on clean_db |

**6. TD-035 closed** in `docs/tech-debt.md`.

## Debugging notes

### `downgrade base` does not drop PostgreSQL ENUM types

When doing `alembic downgrade base` followed by `alembic upgrade head` to get
a clean slate, the migration failed with:

```
sqlalchemy.exc.ProgrammingError: type "accounttype" already exists
```

Cause: Alembic's autogenerated `downgrade()` drops tables but not the
associated PostgreSQL ENUM types (e.g., `accounttype`, `direction`). Those
types persist after `DROP TABLE`.

Resolution: Reset the schema entirely before re-running migrations:

```bash
docker compose exec db psql -U ledger_user -d ledger_db \
  -c "DROP SCHEMA public CASCADE; CREATE SCHEMA public; GRANT ALL ON SCHEMA public TO ledger_user;"
uv run alembic upgrade head
```

### `mapped_column()` argument order matters

`ForeignKey` before `String(3)` raises:

```
sqlalchemy.exc.ArgumentError: 'SchemaItem' expected, got String(length=3)
```

SQLAlchemy's `Column` / `mapped_column` expects the type as the first positional
argument, then SchemaItems. Always write `String(3)` before `ForeignKey(...)`.

## Key Takeaways

### What did I learn?

I learned that PostgreSQL allows a `FOREIGN KEY` to reference any column with a
`UNIQUE` constraint, not just the primary key. `currencies.code` has
`unique=True` but its PK is a UUID, so writing `ForeignKey("currencies.code")`
is valid. The alternative — storing `currency_id UUID` pointing to `currencies.id`
— would have required changing column types and a larger data migration.

I also learned that seeding reference data in `clean_db` is a pragmatic trade-off
for FK constraint support. It blurs the "zero-state" principle slightly, but
USD/EUR/JPY qualify as "reference data" that the system always assumes exists,
so the trade-off is defensible.

### What would I do differently?

I would anticipate earlier that adding a seed to `clean_db` would break tests
that create the same currencies. The cascade of fixes across six test files was
predictable from the design phase, and building the idempotent `_seed_currency`
pattern earlier (or noting the affected files in Step A) would have saved
multiple debugging rounds.

### What surprised me?

`alembic downgrade base` silently leaves ENUM types behind. Tables are dropped,
but `CREATE TYPE accounttype AS ENUM (...)` objects persist in the PostgreSQL
schema. I expected `downgrade base` to be a full clean slate, but it is not.
The fix — `DROP SCHEMA public CASCADE` — is more nuclear than I expected to need
for a routine "wipe and rebuild" workflow.

### What is worth remembering for future goals?

- **Alembic + PostgreSQL ENUM**: After `downgrade base`, always verify that ENUM
  types are gone before re-running `upgrade head`. If in doubt, use
  `DROP SCHEMA public CASCADE; CREATE SCHEMA public;` for a guaranteed clean slate.

- **`mapped_column()` order**: Type argument (`String`, `Integer`, etc.) must
  come before SchemaItems (`ForeignKey`, `CheckConstraint`). This is easy to get
  wrong when copying patterns from documentation.

- **Test seed cascade**: When adding seed data to `clean_db`, grep all test files
  for any helper that creates the same rows (via ORM INSERT or HTTP POST) before
  running pytest. The `_seed_currency` idempotent pattern (SELECT-first) is the
  right template for ORM-layer helpers; the non-conflicting-code approach (GBP/CHF)
  is right for tests whose primary concern is the endpoint behaviour, not the
  specific data.

- **Defense in depth for validation**: App-layer validation (422) + DB FK
  constraint together are the right pattern. The app layer provides user-friendly
  error messages; the FK is the safety net for any bypass. This mirrors the
  existing `_resolve_usd_conversion_rate` pattern in `transaction_service`.
