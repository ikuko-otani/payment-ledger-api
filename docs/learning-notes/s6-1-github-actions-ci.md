## Step C Walkthrough

### C-1: lint job corrections

`ruff check` and `ruff format --check` are independent subcommands:

| Command | Purpose |
|---------|---------|
| `ruff check .` | Lint (unused imports, style violations) |
| `ruff format --check .` | Format check (is code already formatted?) |

`--check` flag makes `ruff format` exit non-zero on diff without modifying files — suitable for CI.

`mypy --strict` enables a bundle of flags including `--disallow-untyped-defs`,
`--disallow-any-generics`, `--warn-return-any`. Explicit in CI even if `pyproject.toml`
already sets `strict = true`, so the CI file is self-documenting.

### C-2: pytest flags

| Flag | Effect |
|------|--------|
| `-v` | Print each test name with PASSED/FAILED |
| `--tb=short` | Compact traceback on failure (good for CI log readability) |

`--tb=long` (default) is useful for local debugging but makes CI logs verbose.

### C-3: build job design

```yaml
build:
  runs-on: ubuntu-latest
  needs: [lint, test]
  steps:
    - uses: actions/checkout@v4
    - name: Build Docker image (health check)
      run: docker build -t payment-ledger-api:ci .
```

`needs: [lint, test]` means lint and test run in parallel; build only starts when both succeed.
`ubuntu-latest` runners include Docker by default — no additional setup needed.

## Key Takeaways

### What did I learn?

- GitHub Actions has two distinct levels of organization: **jobs** (run on separate VMs, can be parallel) and **steps** (run sequentially inside one job). `needs:` controls inter-job dependencies.
- `ruff check` and `ruff format --check` are separate subcommands and must be listed as separate steps. Forgetting `format --check` means formatting regressions silently pass CI.
- `mypy --strict` in the CLI and `strict = true` in `pyproject.toml` are equivalent, but making it explicit in the CI command makes the pipeline self-documenting — a reader does not need to open `pyproject.toml` to understand the strictness level.
- `.coverage` is a binary artifact generated by `pytest --cov` and should always be in `.gitignore`. It was missing since S5-7 when `--cov` was first added.
- `ubuntu-latest` runners include Docker, so `docker build` in CI works with no additional setup steps. No `services:` block needed — the Dockerfile just needs to be valid.

### What would I do differently?

- Add `.coverage` and `htmlcov/` to `.gitignore` at the same time `--cov` is introduced (S5-7), not after the fact. The artifact will appear locally the first time `pytest --cov` is run.
- Verify the existing CI file before reading the Notion "やること" checklist. Notion said "新規作成" but the file already existed — reading the file first would have revealed the duplicate step and missing format check immediately.

### What surprised me?

- The Notion page described the task as creating `ci.yml` from scratch, but the file had already been created in S5-7. The actual work was correction and extension, not creation. This confirms the handoff note: "Notion の『やること』記述は実装より古いことがある."
- The duplicate `ruff check .` step had been silently passing CI since S5-7 — duplicates in YAML are valid syntax and GitHub Actions simply runs the step twice. No warning is emitted.

### What is worth remembering for future goals?

- **CI job graph mental model**: lint and test run in parallel; build (and future deploy) gates on both. Draw the dependency graph before writing `needs:` to avoid accidental sequentialization.
- **`docker build` as a health check**: building without pushing is a low-cost way to catch Dockerfile breakage in CI. Layer cache (`--cache-from`) can be added in S7 when a registry is available.
- **uv cache (`enable-cache: true`)**: the cache key is derived from `uv.lock`. Cache is invalidated automatically when dependencies change — no manual key management needed.
- **S6-2 next**: add `--cov-fail-under` threshold to the pytest step to enforce coverage minimums. TD-013 (async coverage underreporting) means the threshold should be set conservatively at first.
