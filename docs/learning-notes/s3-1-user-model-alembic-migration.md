### Key design: UserRole as PostgreSQL native enum

```python
class UserRole(str, enum.Enum):
    ADMIN = "admin"
    AUDITOR = "auditor"

role: Mapped[UserRole] = mapped_column(
    sa.Enum(UserRole, name="userrole"), nullable=False, default=UserRole.AUDITOR
)
```

The `name="userrole"` argument is what causes Alembic to emit `CREATE TYPE userrole AS ENUM (...)`
in the migration. Without `name=`, the migration may generate an inline VARCHAR check instead.

### Key design: hashed_password naming

`UserCreate` schema accepts `password: str` (plain text).
`User` model stores `hashed_password: str` (bcrypt hash, added in S3-2).
`UserResponse` does not include `hashed_password` — leaking even a bcrypt hash
enables offline brute-force attacks and is considered a security defect.

### mypy Option A config

```toml

    "app.db.session",

    "app.models.transaction",
    "app.dependencies.idempotency",
]
ignore_errors = true
```

New modules are fully type-checked. Existing modules with known-incorrect annotations are
suppressed via overrides rather than fixed — see Key takeaways for rationale.

### ⚠️ Observation: migration renders enum values as uppercase names

The generated migration contains:
```python
sa.Enum('ADMIN', 'AUDITOR', name='userrole')
```

The Python enum has lowercase VALUES (`"admin"`, `"auditor"`), but Alembic rendered uppercase
member NAMES (`'ADMIN'`, `'AUDITOR'`). This discrepancy should be verified in S3-2 when an
actual `User` row is inserted and queried. If SQLAlchemy stores `"AUDITOR"` in the DB but
queries expect `"auditor"`, an enum mismatch error will surface at that point.

## Key takeaways

**What did I learn?**

I learned that registering a new model in `app/models/__init__.py` is what makes Alembic
autogenerate detect it — the model file alone is not enough. `env.py` imports `Base.metadata`
from this package, so any model not imported here is invisible to Alembic at migration time.
I had seen this pattern with Account and Transaction in S2, but this goal made the mechanism
explicit: it is the `__init__.py` import, not the file existence, that matters.

I also learned the difference between PostgreSQL native enum and a CHECK constraint at the
migration level. Native enum creates a separate `pg_type` entry (`CREATE TYPE userrole AS ENUM
(...)`), which means it is a first-class type in the database, visible via `\dT` in psql.
The `name=` argument to `sa.Enum()` is the trigger for this: omitting it may cause Alembic
to skip the `CREATE TYPE` statement entirely.

Setting up mypy for the first time in this project required a deliberate scope decision.
Option A (suppress existing modules with overrides, fully check only new code) was the right
call for a 90-minute goal budget. Option B (fix all existing mypy issues) would have added
40–65 minutes and three separate bug fixes (`get_db` generator type, bare `dict` in
transaction.py, `scalar_one()` cast in balance.py) that are not related to S3-1's purpose.

**What would I do differently?**

I would fix `app.db.session`'s `get_db()` return type annotation in the same sprint as mypy
is introduced, rather than suppressing it with an override. It is a 2-line change
(`-> AsyncGenerator[AsyncSession, None]` + one import) and would remove one override entry
from `pyproject.toml`. Deferring it makes the mypy config harder to reason about over time.

**What surprised me?**

The enum value casing discrepancy in the Alembic migration was unexpected. The Python enum
has lowercase values (`"admin"`, `"auditor"`) but the generated migration rendered uppercase
names (`'ADMIN'`, `'AUDITOR'`). Whether this causes a runtime error depends on how SQLAlchemy
maps Python enum instances to DB values — this was not tested in S3-1 because no test inserts
a User row. It needs to be the first verification step in S3-2.

**What is worth remembering for future goals?**

- Always add new models to `app/models/__init__.py` immediately — Alembic autogenerate
  silently skips models that are not imported into Base.metadata.
- `sa.Enum(EnumClass, name="typename")`: the `name=` argument is not optional for native PG
  enum types. Without it, Alembic may not generate `CREATE TYPE`.
- `hashed_password` in the ORM model, `password` in the request schema: keep this naming
  convention consistent in all future authentication-related models.
- The mypy overrides in `pyproject.toml` are intentional technical debt. When fixing
  `app.db.session` in a future goal, remove the `app.db.session` entry from overrides.
- Verify enum value casing (ADMIN vs admin) in S3-2 before writing service-layer logic.
